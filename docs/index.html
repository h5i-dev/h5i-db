h5i-db is an embedded, versioned time-series database for quantitative workloads and AI agents — a Rust storage kernel with immutable Parquet segments, DataFusion SQL, time travel, and mutations you can preview before they commit.

what it is

ArcticDB's storage model, redesigned — with DuckDB-class queries

Every write is an atomic commit producing an immutable version. Reads never block, writers never clobber each other, and any past state is one manifest read away.

▚Versioned storage

Immutable Parquet segments + per-version manifests, directly addressed by sequence: version reads are O(1), as_of is O(log V). Commits are compare-and-swap on a single HEAD — racing writers get an explicit version_conflict, never last-writer-wins.

◫DataFusion SQL & DataFrames

Full SQL — joins, windows, CTEs — plus time-series operators: a native ASOF JOIN, time_bucket, vwap/wavg, ewma. Storage is time-sorted, so bucketed aggregations stream instead of sorting.

⧗Time travel & snapshots

h5i('trades', 42) reads version 42; RFC3339 reads as-of a commit time; named snapshots pin exact versions across tables — reproducible backtests forever.

▦Manifest pruning

Per-segment time ranges and column min/max live in the version manifest. Narrow time-range queries never open irrelevant files — 48 of 50 segments pruned before I/O in the benchmark, visible per query in --stats.

✓Previewable mutations

--plan runs the full write path but stops before publish: exact affected rows, segments rewritten vs reused, before/after samples. plan apply is a metadata-only CAS. A mutation policy can force agents through review.

⚿Crash-safe by construction

fsync-before-swap, checksums on every object, a manifest hash chain, and tests that kill the writer at every commit step: the old head survives, always. verify --deep re-checks the whole store.

for ai agents

a normal tool, with guardrails that hold

No MCP server, no protocol — agents drive the same CLI and Python API as everyone else. The contract is what makes it agent-ready.

$ h5i-db query market.db "SELECT symbol, vwap(price,size) FROM trades GROUP BY symbol" \
    --format json --max-rows 1000 --timeout 30s        # machine formats + hard limits

$ h5i-db delete-range market.db trades --start 09:30 --end 09:31 --plan
# {"plan_id": "5c41…", "summary": {"rows_affected": 12481, "segments_reused": 127}}
# errors are {code, message, retryable, hint} on stderr; exit codes are stable

$ h5i-db policy set market.db direct_delete=false    # agents must preview; humans can too

{}Machine-readable everything

json / jsonl / csv / arrow output, structured errors with a retryable flag and an actionable hint, stable exit codes (0/2/3/4/5).

◔Resource limits as flags

--max-rows, --timeout, --memory-limit-mb with disk spilling, --read-only. A supervisor can hard-cap any call.

◎Policy-enforceable review

Direct writes and reviewed plans share one write path; the policy chooses which ops may skip preview. Every manifest records execution_mode and the plan hash.

review ui

the approval surface for a versioned database

h5i-db ui market.db serves a loopback-only review surface: pending plans with exact previews, apply/reject, version timeline with audit badges, version diffs, and an SQL scratchpad that shows pruning on every query. Read-only unless started with --allow-mutations.

quickstart

five commands to a versioned database

$ h5i-db init market.db
$ h5i-db create-table market.db trades --like ticks.parquet --time-column ts
$ h5i-db ingest market.db trades ticks.parquet
$ h5i-db query market.db "SELECT * FROM asof_join('trades','quotes','ts','ts','symbol')"
$ h5i-db ui market.db                      # review surface on localhost

# python
>>> db = h5i_db.Database("market.db")
>>> db.sql("SELECT * FROM h5i('trades', 42)").to_pandas()   # time travel
>>> plan = db.plan_delete_range("trades", start, end); plan.apply()
